# レイドバトル風・エンジニアリング系ポケモンライク

## UI要件整理メモ / デザイナー依頼用たたき台

---

## 1. 企画の元アイデア

### 世界観 / コンセプト

* **ポケモンGoのレイドバトルっぽい体験**をベースにする
* 強いボスが出現し、**複数人で協力して倒す**
* 倒したあとに**報酬獲得**と**捕獲チャンス**がある
* 1人では厳しいが、みんなで協力すると勝てる体験を作る
* デモ時間の都合上、**1プレイは3〜3:30程度**で完結させる

### ポケモン要素の置き換え

* “ポケモン”は **プログラミング言語を擬獣化した存在**
* 略称としてポケモンと呼ぶ
* 例:

  * Python
  * Rust
  * JavaScript
  * C
  * TypeScript
    など

### 命名案

* **POlymorphism KErnel MOdule Network**
* 略して **ポケモン**
* エンジニアリング文脈に寄せたネーミング

### ポケモンのプロパティ案

* 名前
* タイプ

  * 動的
  * 静的
  * 関数型
  * 手続き型
  * クラス型
* HP
* 攻撃
* 速度

  * ベンチマークを作って値に反映できると面白い

### アイテム案

* レッドブル
* モンスター
* クッション

### イベント / アクター案

* ユーザーがログインした
* ユーザーが近くの人とマッチングする
* 運営がアイテムを付与した
* ポケモンが攻撃した
* ポケモンが敵を倒して経験値を得た
* リアルタイム対戦 / リアルタイム協力

### やりたいこと

* **みんなで協力してバトルができる**
* QRコードなどで入ってもらう
* Safariで参加できる導線も意識する
* **リアルタイムに共闘してる感**を出したい
* WebSocket / WebRTC / WebTransportなどが候補
* 今回は特に **WebTransportに興味あり**
* Safari非対応なので、デモ時の扱いは考慮が必要

### レイドバトル体験として持ち込みたい要素

* 定刻またはイベント開始でレイドが発生
* バトルは連打寄り
* 一定量攻撃すると技が使える
* 敵を倒すとアイテムがもらえる
* 倒したボスに対して捕獲チャンスがある
* 捕獲は今回は**確率ベースで簡易的**でよい

### ゲーム体験として重視したい点

* **リアルタイムで協力してバトルしてる感**
* 敵HPがみんなの攻撃で減っていくことが見える
* 他プレイヤーの存在感がほしい
* アバターが出て、そこから攻撃が飛んでいく表現は良さそう

### 操作に関する現時点の方針

* 攻撃操作は

  * **画面全体タップ**
  * **攻撃ボタン**
    の両方を比較したい
* 後で採用決定する前提で、両方成立するUI案を持ちたい

### 捕獲について

* 捕獲は確率ベースでよい
* 強いスキル依存にはしない
* あくまでレイド後の気持ち良い締めとして扱う

# 2. 今回のUI設計方針

この企画においてUIで最重要なのは次の4点です。

1. **短時間で始められること**
2. **みんなで戦ってる感があること**
3. **ボスを倒した達成感があること**
4. **報酬・捕獲までテンポよく繋がること**

そのため、UIは単なる画面部品ではなく、下記の元アイデアに紐づいて設計する。

# 9. "AI Slop"風フロントエンドUIの特徴と、プロ品質に引き上げる実践ガイド

## エグゼクティブサマリ

"AI slop"は、生成AIによって大量生産される低品質（低労力・低意味密度）なデジタルコンテンツを指す言葉として広まり、辞書的にも「AIで量産される低品質デジタルコンテンツ」といった定義が明確化されています。
 これをフロントエンドUIに当てはめると、「一見モダンだが、情報階層・一貫性・アクセシビリティ・資産品質（画像/アイコン/文言）のどこかが崩れて"薄っぺらさ"が露呈する」状態として現れやすい、というのが実務的な観測です（本レポートはその"兆候"を、既存の一次情報（Apple/Material/NN/g/W3C）に照らして診断可能な形に整理します）。

プロに見えるUIの根因は、センスよりも「設計判断の体系化」です。具体的には、(a) グリッドと余白のルール、(b) タイポ階層（トークン化されたタイプスケール）、(c) 役割ベースの配色（ロール/トークン）とコントラスト、(d) アイコンセットとストロークの統一、(e) 影（elevation）とモーションの節度、(f) 画像の信頼性（本物感）を、設計→レビュー→実装のワークフローに埋め込むことが重要です。

本レポートでは、ユーザー指定の項目に沿って、(1) 生成AI風UIの共通特徴（箇条書き）、(2) 生成AI風に見える配色パターンと比率（HEX/RGB）、(3) 生成AIっぽい画像の共通要素とアーティファクト、(4) プロ品質に寄せるチェックリスト/ワークフロー（Mermaid図含む）、(5) 生成AIっぽく見えない画像制作の手順、(6) 参考事例の良し悪し分析（画像含む）、(7) 実務テンプレート（配色/プロンプト/レビュー）を提供します。

## 9-1. AI Slop風UIが生まれる背景と診断視点

"slop/AI slop"が問題視される背景には、「生成コストの低下→量産→注意経済（クリック/滞在/広告）での最適化」という構造があります。
 これがUI制作（特にLP/マーケ/ダッシュボードの雛形生成）に入ると、短時間でそれっぽい画面を出せる反面、**"統計的に無難な平均"**に寄りやすく、差別化と品質保証が置き去りになりやすい、というリスクが生まれます。AI生成のWebが似る理由を「学習データ上で最頻の構造（SaaSの典型レイアウト）に収束する」と説明する論考もあり、現象理解としては有用です。

一方、見た目の良さは偶然ではなく、可視化できる原則（グリッド整列、明確な視覚階層、意図した色、全体の一貫性）から生まれる、という整理が確立しています。
 したがって「AI Slopっぽさ」は、センス論ではなく **原則の破れとして診断できます**。加えて、設計の一貫性はデザインシステム（標準の集合）によって維持され、運用には"守らせる仕組み"が必要、という指摘もあります。

このレポートでは診断軸を次の3層に分解します（後続のチェックリストにも反映）。

| 診断レイヤー | 何が崩れると"AIっぽく"見えるか | 参照すべき一次情報の代表 |
|---|---|---|
| ルール層（設計体系） | グリッド/トークン/コンポーネント規約が無く、値が場当たり的 | デザインシステムの考え方、デザイントークン仕様 |
| 表現層（視覚言語） | 階層が曖昧、色や影や角丸が過剰、アイコンが混在 | 視覚デザイン原則、Material/Appleのスタイルガイド |
| 検証層（信頼/アクセシビリティ） | コントラスト不足、タップ領域不足、画像が不自然/汎用的で信用を損なう | WCAG、タッチターゲット、写真の信頼性研究 |

## 9-2. 生成AI風UIと画像の共通特徴

### 生成AI風UIの共通ビジュアル特徴（事例つき・箇条書き）

以下は「AIが作ったように見える」UIで頻出する兆候を、観察可能な形に分解したチェック観点です。各項目は、視覚デザインの基本原則（グリッド・階層・一貫性・色の意図）に反していることが多く、結果として"薄い/それっぽいだけ"に見えます。

#### レイアウト（構造）

* 主役のない「カードの寄せ集め」：Bento/カードが均等に並び、最重要CTAや見出しが埋もれる（例：ヒーロー直下が同サイズカード3〜6枚で同価値に見える）。
* "SaaS LP平均"テンプレ：中央寄せヒーロー＋3列アイコン説明＋ロゴ行＋FAQという定型がそのまま出る（例：差別化要素が「高速」「安全」「簡単」など抽象語）。
* 画面幅ごとの規律不足：ブレークポイントで余白・カラム数・文字サイズが飛び、リズムが崩れる（例：PCは過疎、SPは過密）。

#### タイポグラフィ

* 階層が"フォントサイズの乱数"：見出し/本文/ラベルの役割が固定されず、画面ごとにサイズ・ウェイトが変わる（例：H2が24px→28px→22pxと揺れる）。
* 文字詰め/行間が未調整：とくに英字のtracking、本文のline-heightが不自然で"テンプレ感"が出る（例：本文が詰まり過ぎて読みにくい）。
* グラデーション文字や極太見出しの濫用：アクセント表現が多すぎ、情報強調が飽和して「全部重要」に見える。
* "UI内に画像化テキスト"が混じる：後述の生成画像由来で、UIテキストがビットマップになり、可読性とアクセシビリティを損なう（WCAGは原則として画像文字を避ける）。

#### アイコン

* セット混在：アウトライン/フィル/2色/3Dが同一画面に混ざり、ストローク太さや角の丸みが揃わない（例：ナビは線アイコン、カード内は塗りアイコン）。
* "解像度の違い"が出る：SVGとラスタが混在し、輪郭が甘いアイコンが混ざる。
* 状態表現が曖昧：選択/非選択、hover/pressedで形が変わるなど、意味が統一されない（可変軸のあるアイコンでは軸で状態を表現可能）。

#### 余白とグリッド

* 8の倍数/4の倍数のリズムがない：paddingやgapが 13px・19pxのように散らばり、"手が入ってない感"が出る（例：カードごとに内側余白が異なる）。
* セクション間の呼吸が不均一：上だけ広い/下だけ詰まるなど、スクロール時のテンポが乱れる（視覚階層と連動しない）。
* ベースラインのズレ：テキストの整列線が揃わず、微妙なガタつきで安っぽく見える。

#### グラデーション/エフェクト

* "紫〜青の巨大ブラー"が背景に鎮座：視線誘導ではなく装飾として居座り、本文コントラストや可読性を落とす（例：背景ノイズの上に薄いグレー本文）。
* Glassmorphismの乱用：透過カード＋背景ブラー＋細い白枠が多すぎ、階層が逆に読めなくなる。Glassmorphismは原理上、背景が複雑だとアクセシビリティ課題が出やすいと整理されています。
* 影が"なんとなく"付いている：どの要素が浮くべきかのルールがなく、全部が浮いて見える（elevationの意図が崩壊）。

#### シャドウ/ボタン形状/タップ領域

* 角丸が場当たり：ボタンはピル形状、カードは中途半端な角丸、モーダルは別の角丸…とスケールがない（Materialは角丸スケールを用意）。
* 「Primaryが多すぎる」：同一画面に強いCTAが複数あり、意思決定ができない（Appleのボタン指針でも"目立たせるボタン数"は抑える思想が示される）。
* タップしづらい：ヒット領域が小さい。Appleは一般に44×44pt以上のヒット領域を推奨し、Material側は多くのプラットフォームで48×48dp以上のタッチターゲットを推奨します。

#### アニメーション

* "全部フェード/全部スライド"症候群：同じモーションが多用され、情報の意味を増やさない（動きは関係性や遷移理解を助けるためのもの）。
* 減らす設定に配慮しない：OSの「Reduce Motion」に追従せず、疲労や酔いの原因になる（Appleはモーション削減設定への配慮を求める）。
* Easing/Durationの統一がない：同じ種類の遷移が画面ごとに"別物"に見える（Material 3はeasing/durationの考え方を体系化）。

#### 一貫性の欠如（総合）

* コンポーネントの"意味"が固定されない：同じ見た目が別の意味で使われ、学習が起きない（一貫性と標準への準拠はユーザー学習を助ける）。
* 文章（マイクロコピー）が汎用/抽象：具体性がなく、信頼が積み上がらない（例：「Seamless」「Boost productivity」だけが並ぶ）。

上記の多くは、**「グリッド整列」「視覚階層」「色の意図」「一貫性」**という、見た目を"良く見せる"ための基本原則に直結しています。

### 生成AIっぽく見える画像の共通要素とアーティファクト例

生成画像が"AIっぽい"と判定される要因は、（A）人間の身体や物理の破綻、（B）テキストの破綻、（C）質感・照明の不自然さ、（D）局所的な形状/テクスチャ破綻、などのカテゴリに整理できます。

#### 人物写真風

* 手指の破綻（本数、関節、指同士の融合、余計な指）：拡散モデルの典型的失敗として論文レベルで扱われ、修復フレームワークも提案されています。
* 皮膚が"プラスチック"っぽい／毛髪が"ふわふわし過ぎる"：質感が均質化し、微細な不規則性が足りない（スタイル/質感アーティファクトとして指摘）。
* 照明の整合性が変：顔と背景、物体の影の向きが一致しない（照明/影の不整合は偽造検知でも典型手掛かり）。

#### イラスト/3D風

* 線幅の揺れ、エッジのにじみ：アイコンやUIイラストとして使うと"線の仕様"に乗らず違和感が出る。
* ロゴ/文字を描かせると崩れる：テキストを含む画像生成は依然難しく、読めない文字列になりやすいことが明示されています。

#### 抽象背景/テクスチャ

* 反復パターン（タイルの繰り返し）や局所的な歪み：とくにグラデ背景や"生成っぽいノイズ"がUIの背後にあると、コンテンツより背景が主張する。
* 目的不明の擬似ディテール：それっぽいが意味がない装飾が増え、UI密度が上がる。

実務上は「UI上の画像は"情報"として機能させる」ことが重要で、意味のない写真/イメージは注視されにくい（とくに汎用ストック）という視線計測の知見があります。

## 9-3. 生成AI風に見えやすい配色パターンと、避けるための色設計

### 生成AI風に見える配色パターン・色比率・具体パレット

まず前提として、プロダクトUIは「好きな色を置く」のではなく、**役割（role）**で色を割り当て、組み合わせを誤るとコントラスト（視認性）を壊す、という考え方が一次情報で明確にされています（Materialのcolor rolesは、誤った組み合わせがアクセシビリティ上のコントラストを壊す可能性を明示）。
 さらにW3CのWCAGは、通常テキストで最低4.5:1、大きなテキストで最低3:1のコントラストなどを要求し、UIコンポーネントの非テキスト要素にも3:1を求めます。

その上で、「生成AI風に見えがち」な配色は、(a) 紫〜青グラデ、(b) ネオン系アクセント、(c) 暗背景＋薄い境界線＋透明感に偏りやすい、という観察があります。

以下は「AIっぽさが出やすい」代表パターンです（※目的は模倣ではなく、レビュー時に「この方向だとAI slopに見えやすい」を特定するための参照）。

| パターン名 | 色比率の典型 | 役割ごとの例（HEX / RGB） | "AIっぽさ"が出る理由（実務的） |
|---|---|---|---|
| 紫〜青のグロウ暗背景（ネオンサイバー） | 背景70%・サーフェス20%・アクセント10% | BG #0B1020 (11,16,32) / グロウ #7C3AED (124,58,237) 〜 #22D3EE (34,211,238) / Surface #111827 (17,24,39) / Text #E5E7EB (229,231,235) / Border #243041 (36,48,65) | "紫グラデSaaS"に収束しやすいという指摘があるテーマで、背景装飾が主役になりやすい。 |
| 白背景＋パステルブラー（リキッド/ジェリー） | 背景60%・白カード30%・アクセント10% | BG #F7F8FC (247,248,252) / Blob #A78BFA (167,139,250), #60A5FA (96,165,250), #5EEAD4 (94,234,212) / Card #FFFFFF (255,255,255) / Text #111827 (17,24,39) / Subtext #6B7280 (107,114,128) | "とりあえず今っぽい"の装飾で差別化が弱く、情報階層の設計不足を隠す方向に働きやすい。 |

| ネオングリーン差し（ダーク×蛍光） | 背景75%・サーフェス20%・ネオン5% | BG #0B0F14 (11,15,20) / Surface #111827 (17,24,39) / Primary #22C55E (34,197,94) / Warning #F59E0B (245,158,11) / Text #E5E7EB (229,231,235) | アクセント色が強すぎると"デモ感"が出て、階層が色だけに依存しがち（色だけに頼らないのが原則）。 |

### "AI slop配色"から抜けるための設計ルール

* 色は"役割名"で管理し、見た目で命名しない（例：color.text.primary、color.surface.1）。Materialはロールベースの色体系を持ち、Appleも環境（ライト/ダーク等）に自動適応するシステムカラーの利用を推奨しています。
* **アクセントは原則1色（＋状態色）**に絞り、濃淡はトーン（明度差）で作る。色の使い過ぎは情報の優先度を壊します。
* コントラストはWCAGを"デザインQAのゲート"にする（通常テキスト4.5:1、非テキスト要素3:1など）。
* ダークモード/外観差に耐える配色を前提にする（Appleはダークモードや最小コントラストのチェックを求める）。

## 9-4. プロのデザイナー風UIにするためのワークフローと改善手法

### 実践的チェックリストとワークフロー（ワイヤー→ハイファイ→プロトタイプ→レビュー）

デザイン工程は「きれいな完成画を作る」よりも、「不確実性を段階的に潰す」ことが目的です。低忠実度/高忠実度プロトタイプは目的と観察したい学習に応じて使い分けるべき、という整理があります。
 またデザインレビュー（クリティーク）は"審判"ではなく、目標に照らして改善するための構造化された会話である、という定義が明確です。

```
要件・制約の明確化
    ↓
情報設計 / IA
    ↓
ワイヤーフレーム（低忠実度）
    ↓
ビジュアル方針決定（トークン/グリッド/タイプ）
    ↓
ハイファイUI
    ↓
プロトタイプ
    ↓
デザインレビュー
    ↓
アクセシビリティ/可用性チェック
    ↓
実装ハンドオフ
    ↓
UI QA / 実機確認
```

### 改善手法を"実装単位"に落とす（何を直すとプロに見えるか）

#### グリッドと余白

* 8dp/8ptベースのグリッドで、レイアウト・コンポーネント・間隔を揃える（小要素は4dpも併用）。Androidの設計ガイドラインは8dpグリッドと小要素の4dp整列を明示します。
* 余白は"数値"ではなく"意味"で持つ：space.section、space.cardPaddingのようにトークン化して揺れを止める。

#### タイポ階層

* タイプスケールを固定し、見出し/本文/ラベルの用途を"役割"で決める（Materialはタイプスケールと役割を定義）。
* AppleプラットフォームではDynamic Type等の仕組みを活かし、可読性・階層・一貫性を確保する思想が示されます。

#### アイコンセット統一

* 1つの体系（Material Symbols / SF Symbols等）を選び、太さ・塗り・光学サイズ（optical size）などの軸で状態やサイズに整合を持たせる。Material Symbolsは可変軸（fill/weight/grade/opsz）を備える。
* Apple側もSF Symbolsのレンダリングモード（monochrome/hierarchical/palette/multicolor）などを提供し、統一的なアイコン表現が可能です。

#### 影（elevation）と形（shape）

* 角丸はスケールで管理し、コンポーネントに政策的に割り当てる（Material 3のshapeスケール）。
* 影は"なんとなく"付けず、階層・分離が必要な時だけ使う（Material 3のelevation/ボタン指針では影の過剰使用を避ける意図が見える）。

#### モーション

* モーションは関係性と結果を伝えるために使う（Materialのモーション原則）。
* OSの"動きを減らす"設定に追従し、不要なアニメーションを抑制する（AppleのMotionガイドライン）。
* easing/durationをUI全体で統一する（Material 3のeasing/duration体系）。

#### アクセシビリティ（最低限の"プロの作法"）

* コントラスト：通常テキスト4.5:1、非テキスト要素3:1（WCAG）。
* タッチターゲット：44×44pt（Apple）、48×48dp（Material側推奨）。
* "色だけ"で状態や意味を伝えない。

#### 運用（AI slop化を防ぐ最後の砦）

* デザインシステムには"エンフォーサー（守らせる役割/仕組み）"が必要、という指摘があり、レビューゲートやリンター（トークン未使用の検知）を組み込むのが現実的です。

## 9-5. 生成AIっぽく見えない画像制作プロセス

### 撮影/素材選定/編集/生成AI利用時のプロンプト設計とポストプロセス（ステップ別）

ここでのゴールは「AIを隠す」ではなく、UIの信頼性（リアリティ/一貫性/目的適合）を毀損しない画像を作ることです。

### ステップ一：画像の役割を定義する（UI内での"仕事"を決める）

* 目的を1つに絞る：信頼（人物/現場）・理解（図解/スクショ）・感情（ブランド世界観）・操作支援（チュートリアル）など。
* KPIを決める：例）LPなら「CTA到達率」、オンボーディングなら「完了率」。
* UI上の読み順に合わせる（視覚階層の一部として扱う）。

### ステップ二：撮影する場合（最も"AIっぽさ"を避けやすい）

* 本物の文脈を写す：実際の職場/製品/ユーザー環境。
* ライティングを統一：同一プロダクト内で色温度・影の方向・コントラストを揃える（混ざると"合成感"が出る）。
* 仕上げで粒状感（微細ノイズ）を整える：過度なスムージングは"生成っぽい質感"に寄せやすい。

### ステップ三：素材サイトで選ぶ場合（"汎用ストック"回避が要点）

* "概念を表すだけの人物"を避ける：汎用的な人物ストックは注視されにくいという研究知見に注意。
* 画像をコンテンツとして扱う：何を伝える写真かが説明できないなら外す。
* 同一トーンで揃える：撮影者/レンズ/編集がバラバラだと"寄せ集め感"が出る。

### ステップ四：生成AIを使う場合（"UIに溶け込む品質"に寄せる設計）

テキスト生成は依然として難しく、「読めない文字列（gibberish）」になりやすいことが明示されています。したがって、UIに載る文字は画像内に生成させず、UI側で組版するのが安全です。

#### プロンプト設計（原則）

* 文章は「被写体（subject）＋文脈（context）＋スタイル（style）＋制約（no text等）」で構造化する。
* "ブランドらしさ"は抽象形容詞より、具体要素（素材、光、焦点距離、構図、色温度、背景の種類）で指定する。
* 禁止事項（例：文字、ロゴ、余計な指、歪み）を明示し、複数回生成→選別→編集を前提にする。

#### ポストプロセス（実務で効く順）

* 人体（特に手指）を最優先で検査（破綻は強い違和感として残る）。
* 文字要素が紛れたらUI側で置き換え（生成文字は崩れやすい）。
* 色調整：UIのカラートークンに合わせて彩度/明度を寄せ、画像だけ浮かせない。
* シャープ/ノイズ：過度なスムーズさを抑え、実写素材の粒状感に寄せる。

#### 透明性（任意だが推奨）

生成AI利用の有無を制作側で管理したい場合、来歴情報（Content Credentials）を扱う仕組みが標準化されつつあり、C2PAがオープン標準を提供しています。

## 9-6. 参考事例（良し悪し分析）

### AppleプラットフォームのシステムUI（Human Interface Guidelinesの思想）

* **良い点：** 可読性と階層（タイポ）、適応的レイアウト（safe areas/margins）、モーション節度、タップしやすさ（44×44pt）など、"プロ品質の下限"を具体ルールとして持つ。
* **注意点：** ガイドをそのまま適用すると"Appleっぽい既視感"が出るため、差別化は情報設計とコンテンツ（コピー/画像）側で作るのが安全。

### Material Design 3（構成要素がトークン/スケールとして整備）

* **良い点：** タイプスケール、shapeスケール、elevation、motionなどが体系化され、設計判断が"個人の勘"から切り離される。
* **注意点：** デフォルト寄りだと"Materialアプリ感"が出るため、ブランド差は配色（ロール設計）とコンテンツで作る（ロールの組み合わせ誤りでコントラストを壊さない）。

### IBMのCarbon Design System（企業UIの現実解）

* **良い点：** スペーシングスケールを2/4/8の倍数で整備し、トークンとして運用することで"揺れ"を止めている。
* **注意点：** 堅牢な分、情緒・遊びを入れるときは"例外ルール"を別途設けないと崩れやすい。

### AtlassianのAtlassian Design System（可読性と運用の両立）

* **良い点：** タイポトークン運用、最小フォントサイズの指針など、実装で破綻しやすい部分を明示的にガード。
* **注意点：** 密度の高いプロダクト向きの思想が強いため、ブランド表現は色・イラスト・コピーで設計する。

### StripeのStripe Apps UI（コンポーネント＋パターンで品質を担保）

* **良い点：** コンポーネントと推奨パターンを提供し、カスタムスタイルを意図的に制限して"高品質で一貫した体験"を作る方針が明文化されている。
* **注意点：** 強いブランド表現はやりにくい。代わりに情報設計とコンテンツ品質で差を作る。

### AI生成テンプレ的LP（"紫グラデ＋ブラー＋3列説明"系）

* **良い点：** 短時間でそれっぽい出力が得られる。
* **悪い点：** 同型の表現が量産されやすく、差別化・信頼性・可読性の設計が薄くなる。紫グラデやテンプレ構造に収束するという観察もある。
* **改善の方向：** 配色を役割ベースで再設計、グリッド/タイプ/アイコン/影/モーションをトークン化し、画像を"情報"として選ぶ（汎用ストック/不自然生成を避ける）。

## 9-7. 実務で使えるチェックリストと短いテンプレート

### すぐ使えるUIチェックリスト（レビュー用・短縮版）

* **視覚階層：** 最重要アクションが1つ決まっている（色/サイズ/配置/余白で説明できる）。
* **グリッド：** 主要な余白/サイズ/gapが8（または4）の倍数に揃っている。
* **タイポ：** 役割（見出し/本文/ラベル）ごとにトークンが固定され、画面間でブレていない。
* **アイコン：** 1つの体系を採用し、stroke/filled/opszなどの軸が一貫している。
* **影/角丸：** スケールがあり、影は必要時のみ（全部浮かない）。
* **アクセシビリティ：** コントラスト（4.5:1/3:1）とタップ領域（44pt/48dp）を満たす。
* **画像：** 意味がある（説明できる）／汎用ストックや破綻生成が混ざっていない。

### 配色テンプレ（トークンひな形）

```
# Color tokens (role-based)
color.bg.default
color.bg.subtle
color.surface.1
color.surface.2
color.border.default
color.text.primary
color.text.secondary
color.primary.default
color.primary.on
color.primary.container
color.primary.onContainer
color.state.success
color.state.warning
color.state.danger

# Ratio guideline (starting point)
# bg 60–75% / surfaces 20–35% / accents 5–10%
```

（役割での色管理と組み合わせの注意は、Materialのcolor rolesが強調する考え方です。）

### 生成AI画像プロンプトテンプレ（"UIに溶け込む"前提）

```
[GOAL]
- Use as: (hero image / feature illustration / background texture)
- Message: (what the image must communicate in 1 sentence)

[SUBJECT + CONTEXT]
- Subject: ...
- Environment: ...
- Action: ...

[STYLE (concrete, not abstract)]
- Medium: photograph / vector illustration / 3D render
- Lighting: soft daylight / studio softbox / overcast
- Lens: 35mm / 50mm, shallow depth of field (if photo)
- Color mood: match UI palette, low saturation, neutral background

[CONSTRAINTS]
- no text, no logos, no watermarks
- anatomically correct hands, no extra fingers
- clean edges, no artifacts, no distorted objects
- leave negative space for UI copy

[OUTPUT]
- aspect ratio: 16:9
- high detail, consistent lighting
```

（プロンプトを「subject/style」などの構造で扱い、失敗モードを踏まえて反復する、という観点はHCI研究でガイドラインとして整理されています。なお画像内テキストは崩れやすい課題が明示されているため、UI側で組版するのが安全です。）
